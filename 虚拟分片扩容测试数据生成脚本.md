## 一、脚本说明

### 1.1 目的

生成符合基因法规则的测试数据，用于验证虚拟分片扩容方案的正确性。

### 1.2 数据特点

- **订单号生成**：遵循基因法，订单号的后3位嵌入用户ID的基因位
- **数据分布**：确保数据均匀分布在8个物理分片（2库×4表）
- **关联数据**：生成订单、购票人订单、订单记录三类关联数据
- **可验证性**：可通过虚拟分片ID验证数据正确路由

### 1.3 生成数据量

- 每个物理分片生成约1000条订单
- 总计约8000条订单及对应的关联数据

## 二、核心函数

### 2.1 基因法订单号生成函数

```sql
DELIMITER $$

-- =====================================================
-- 函数：generate_order_number_with_gene
-- 功能：根据用户ID生成包含基因位的订单号
-- 参数：
--   p_user_id: 用户ID
--   p_sequence: 序列号（模拟雪花ID的序列部分）
-- 返回：订单编号（BIGINT）
-- =====================================================
CREATE FUNCTION generate_order_number_with_gene(
    p_user_id BIGINT,
    p_sequence INT
)
RETURNS BIGINT
DETERMINISTIC
BEGIN
    DECLARE v_gene_bits INT;
    DECLARE v_timestamp BIGINT;
    DECLARE v_order_number BIGINT;
    
    -- 1. 提取用户ID的后3位作为基因位（2库×4表=3位基因）
    SET v_gene_bits = p_user_id & 7;  -- 7 = 0b111
    
    -- 2. 生成时间戳部分（使用当前时间戳的秒数）
    -- 简化版：用秒级时间戳，实际生产环境用雪花算法
    SET v_timestamp = UNIX_TIMESTAMP();
    
    -- 3. 生成订单号
    -- 结构：[时间戳(40位)][数据中心(5位)][机器ID(5位)][序列号(10位)][基因(3位)]
    -- 简化版：[时间戳(40位)][序列号(21位)][基因(3位)]
    SET v_order_number = (v_timestamp << 24)     -- 时间戳左移24位
                       | (p_sequence << 3)        -- 序列号左移3位
                       | v_gene_bits;             -- 基因位（后3位）
    
    RETURN v_order_number;
END$$

DELIMITER ;
```

### 2.2 计算虚拟分片ID函数

```sql
DELIMITER $$

-- =====================================================
-- 函数：calculate_logical_shard_id
-- 功能：计算虚拟分片ID
-- 参数：p_sharding_key - 分片键（订单号或用户ID）
-- 返回：虚拟分片ID（0-1023）
-- =====================================================
CREATE FUNCTION calculate_logical_shard_id(p_sharding_key BIGINT)
RETURNS INT
DETERMINISTIC
BEGIN
    DECLARE v_gene_bits INT;
    DECLARE v_table_index INT;
    DECLARE v_database_index INT;
    DECLARE v_physical_shard_index INT;
    DECLARE v_virtual_offset INT;
    DECLARE v_logical_shard_id INT;
    
    -- 步骤1：提取基因位（后3位）
    SET v_gene_bits = p_sharding_key & 7;  -- 0b111 = 7
    
    -- 步骤2：解析基因位，计算物理分片索引
    SET v_table_index = v_gene_bits & 3;        -- 后2位：表索引（0-3）
    SET v_database_index = (v_gene_bits >> 2) & 1;  -- 第3位：库索引（0-1）
    SET v_physical_shard_index = v_database_index * 4 + v_table_index;
    
    -- 步骤3：计算虚拟偏移（0-127）
    SET v_virtual_offset = ABS(p_sharding_key) % 128;
    
    -- 步骤4：计算虚拟分片ID
    SET v_logical_shard_id = v_physical_shard_index * 128 + v_virtual_offset;
    
    RETURN v_logical_shard_id;
END$$

DELIMITER ;
```

### 2.3 获取物理库索引函数

```sql
DELIMITER $$

-- =====================================================
-- 函数：get_database_index
-- 功能：根据分片键计算物理库索引
-- 参数：p_sharding_key - 分片键
-- 返回：数据库索引（0-1）
-- =====================================================
CREATE FUNCTION get_database_index(p_sharding_key BIGINT)
RETURNS INT
DETERMINISTIC
BEGIN
    DECLARE v_gene_bits INT;
    DECLARE v_database_index INT;
    
    SET v_gene_bits = p_sharding_key & 7;  -- 后3位
    SET v_database_index = (v_gene_bits >> 2) & 1;  -- 第3位
    
    RETURN v_database_index;
END$$

DELIMITER ;
```

### 2.4 获取物理表索引函数

```sql
DELIMITER $$

-- =====================================================
-- 函数：get_table_index
-- 功能：根据分片键计算物理表索引
-- 参数：p_sharding_key - 分片键
-- 返回：表索引（0-3）
-- =====================================================
CREATE FUNCTION get_table_index(p_sharding_key BIGINT)
RETURNS INT
DETERMINISTIC
BEGIN
    DECLARE v_gene_bits INT;
    DECLARE v_table_index INT;
    
    SET v_gene_bits = p_sharding_key & 7;  -- 后3位
    SET v_table_index = v_gene_bits & 3;   -- 后2位
    
    RETURN v_table_index;
END$$

DELIMITER ;
```

## 四、使用方法

### 4.1 创建函数和存储过程

**⚠️ 重要说明**：
- **只需在 `damai_order_0` 数据库中创建一次即可**
- 存储过程会自动在两个库（`damai_order_0` 和 `damai_order_1`）中插入数据
- 函数可以通过完全限定名跨库调用（如 `damai_order_0.calculate_logical_shard_id()`）

```sql
-- 1. 连接到 damai_order_0 数据库（只需在这个库中创建）
USE damai_order_0;

-- 2. 执行上述所有函数和存储过程的创建脚本
-- （复制粘贴 "二、核心函数" 和 "三、数据生成存储过程" 中的所有SQL）

-- 3. 验证创建成功
SHOW FUNCTION STATUS WHERE Db = 'damai_order_0';
SHOW PROCEDURE STATUS WHERE Db = 'damai_order_0';
```

**为什么不需要在两个库都创建？**

1. 存储过程中使用了完全限定表名：
   ```sql
   INSERT INTO damai_order_0.d_order_0 ...
   INSERT INTO damai_order_1.d_order_0 ...
   ```

2. 函数支持跨库调用：
   ```sql
   SELECT damai_order_0.calculate_logical_shard_id(order_number) ...
   ```

3. 只维护一份代码，避免不一致

### 9.4 常见问题

**Q: 是否需要在两个数据库（damai_order_0 和 damai_order_1）都创建函数和存储过程？**
A: **不需要！** 只需在 `damai_order_0` 中创建一次即可。存储过程会自动向两个库插入数据，函数可以通过完全限定名跨库调用。

**Q: 为什么数据量不是完全均等的？**
A: 因为用户ID是连续递增的，基因位会循环（0,1,2,3,4,5,6,7），但模128的结果会有随机性，导致略有偏差。

**Q: 如何确保订单号和用户ID的基因位相同？**
A: `generate_order_number_with_gene` 函数会从用户ID中提取后3位作为基因位，并嵌入到订单号的后3位中。

**Q: 生成数据速度慢怎么办？**
A: 减少每批生成的数量，或者在存储过程中添加批量插入优化。

**Q: 如果在两个库都创建了会怎样？**
A: 不会报错，但不推荐。因为需要维护两份代码，修改时容易漏改一个库，造成逻辑不一致。
