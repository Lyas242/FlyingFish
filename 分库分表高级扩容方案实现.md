## 方案1：虚拟分片 + 路由表（⭐⭐⭐⭐⭐ 推荐）

> **📖 完整方案文档**：[虚拟分片扩容方案完整说明.md](./虚拟分片扩容方案完整说明.md)
> 
> 本节仅提供概览，详细的实现细节、代码示例、扩容流程请查看完整文档。

### 1.1 核心思想（基于基因位设计）

⚠️ **重要**：本方案完全保留订单号的基因位设计！

```
第1层：订单号基因位（固定不变）
  订单号生成：2库×4表 → 3位基因（000-111）→ 8个原始物理分片
                          ↓
第2层：虚拟分片映射（1024个）
  8个原始物理分片 × 128 = 1024个虚拟分片
  • 每个原始物理分片对应128个虚拟分片
  • 通过userId嵌入的虚拟分片细分位（7位），精确定位到128个虚拟分片之一
  • 保证：orderNumber和userId映射到完全相同的虚拟分片（基于基因法）
                          ↓
第3层：路由表映射（动态可变）
  1024个虚拟分片 → 路由配置表 → 实际物理库表
                    ┌──────────┬────────┬────────┐
                    │逻辑分片ID│物理库  │物理表  │
                    ├──────────┼────────┼────────┤
  │   0-127  │  db_0  │table_0 │ ← 原始物理分片0
  │  128-255 │  db_0  │table_1 │ ← 原始物理分片1
                    │   ...    │  ...   │  ...   │
  │  512-639 │  db_1  │table_0 │ ← 原始物理分片4
                    └──────────┴────────┴────────┘
```

**关键优势**：
- ✅ **保留基因位**：order_number和user_id查询结果一致
- ✅ **细粒度扩容**：1024个虚拟分片，可逐个迁移
- ✅ **代码无需改动**：订单号生成逻辑保持不变
- ✅ **支持灰度回滚**：修改路由表即可，秒级切换

**三层映射的意义**：
1. **基因位层**：保证查询一致性（订单号和用户ID）
2. **虚拟分片层**：提供扩容的灵活性（细粒度）
3. **路由表层**：实现动态映射（可配置）

### 1.2 算法原理

> **📖 详细算法实现**：请查看 [虚拟分片扩容方案完整说明.md](./虚拟分片扩容方案完整说明.md) 第二章和第四章

**核心步骤**：
1. **提取基因位**：从订单号/用户ID提取后10位（物理基因3位 + 虚拟细分7位）
2. **解析虚拟分片**：后3位确定物理分片索引（0-7），前7位确定虚拟分片内索引（0-127）
3. **计算虚拟分片ID**：物理分片索引 × 128 + 虚拟分片内索引
4. **查询路由表**：获取虚拟分片对应的实际物理库表

**关键配置**：
```yaml
sharding:
  original:
    database-count: 2  # 必须与订单号生成时一致
    table-count: 4     # 必须与订单号生成时一致
```

**✅ 核心优势**：

基于基因法的完美一致性设计：
- orderNumber嵌入userId的10位信息（物理基因3位 + 虚拟细分7位）
- orderNumber和userId的后10位完全相同
- 因此映射到**完全相同**的虚拟分片

**迁移灵活性**：
```
✅ 单个虚拟分片迁移：虚拟分片56 → 迁移到 d_order_4
✅ 批量虚拟分片迁移：虚拟分片0-63 → 迁移到 d_order_4
✅ 灵活调整：根据负载情况自由调整迁移策略
```

详细说明请查看：[虚拟分片扩容方案完整说明.md](./虚拟分片扩容方案完整说明.md)

### 1.3 优缺点总结

#### ✅ 优点

1. **完美查询一致性**：orderNumber和userId映射到完全相同的虚拟分片（基于基因法）
2. **极细粒度扩容**：1024个虚拟分片，单个虚拟分片迁移粒度1/1024（约0.1%数据）
3. **API简洁统一**：订单号生成方法自动支持虚拟分片，只需3个参数
4. **灵活迁移策略**：可单个、批量或范围迁移，根据负载情况自由调整
5. **灰度和回滚**：可逐步切换虚拟分片，观察效果，秒级回滚
6. **支持热更新**：路由表变更后实时生效，无需重启服务
7. **多次扩容**：支持从8→16→32...多次扩容，每次都是增量迁移

#### ❌ 缺点

1. **维护路由表**：需要额外维护1024条路由映射记录
2. **内存占用**：需要缓存路由映射（约50KB，可忽略不计）
3. **实现复杂度中等**：需要理解基因位设计，实现路由管理器、迁移执行器等组件
4. **仍需数据迁移**：虽然粒度极细，但扩容时仍需迁移数据（无法避免）

#### 💡 关键设计理念

**三层映射的核心价值**：
1. **基因位层**（第1层）：订单号固化的基因，保证查询一致性
2. **虚拟分片层**（第2层）：应用层抽象，提供扩容的灵活性
3. **路由表层**（第3层）：动态映射，实现物理分片的灵活调整

**为什么需要虚拟分片？**
- 订单号嵌入10位userId信息（物理基因3位 + 虚拟细分7位）
- 虚拟分片将8个物理分片扩展到1024个逻辑分片（×128倍）
- 扩容时可以按单个虚拟分片粒度迁移（1/1024），风险最低
- 完美一致性：orderNumber和userId的后10位相同，映射到完全相同的虚拟分片

## 方案2：一致性哈希（⭐⭐⭐⭐）

### 2.1 核心思想

```
一致性哈希环（0 ~ 2^32-1）
         ┌─────────────────┐
         │                 │
     Node1(虚拟节点)    Node2(虚拟节点)
         │                 │
         │   Key1 → 顺时针查找最近节点
         │   Key2 → 
         │                 │
         └─────────────────┘
         
物理节点：db_0.table_0, db_0.table_1, ...
每个物理节点创建 100-200 个虚拟节点，确保负载均衡
```

**优势**：
- ✅ 扩容时只影响相邻节点的数据
- ✅ 自动负载均衡（通过虚拟节点）
- ✅ 节点故障影响范围可控

### 2.2 核心实现

#### 2.2.1 一致性哈希环管理器

```java
package com.damai.shardingsphere;

import lombok.Data;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.security.NoSuchAlgorithmException;
import java.util.*;
import java.util.concurrent.ConcurrentSkipListMap;

/**
 * 一致性哈希环管理器
 */
@Slf4j
@Component
public class ConsistentHashingManager {
    
    /**
     * 虚拟节点数量（每个物理节点对应的虚拟节点数）
     * 建议 100-200，越多负载越均衡，但初始化越慢
     */
    private static final int VIRTUAL_NODE_COUNT = 150;
    
    /**
     * 哈希环：使用TreeMap自动排序
     * Key: 哈希值
     * Value: 物理节点信息
     */
    private final TreeMap<Long, PhysicalNode> hashRing = new TreeMap<>();
    
    /**
     * 物理节点列表
     */
    private final List<PhysicalNode> physicalNodes = new ArrayList<>();
    
    /**
     * 初始化哈希环
     * 
     * @param databases 数据库列表
     * @param tablesPerDatabase 每个数据库的表数量
     */
    public void initialize(List<String> databases, int tablesPerDatabase) {
        physicalNodes.clear();
        hashRing.clear();
        
        // 1. 创建物理节点列表
        for (String database : databases) {
            for (int tableIndex = 0; tableIndex < tablesPerDatabase; tableIndex++) {
                PhysicalNode node = new PhysicalNode(database, "d_order_" + tableIndex);
                physicalNodes.add(node);
            }
        }
        
        // 2. 为每个物理节点创建虚拟节点
        for (PhysicalNode node : physicalNodes) {
            addPhysicalNode(node);
        }
        
        log.info("一致性哈希环初始化完成：{} 个物理节点，{} 个虚拟节点",
            physicalNodes.size(), hashRing.size());
    }
    
    /**
     * 添加物理节点到哈希环
     */
    public void addPhysicalNode(PhysicalNode node) {
        for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
            // 虚拟节点标识：物理节点名称 + 虚拟节点编号
            String virtualNodeKey = node.getIdentifier() + "#VN" + i;
            long hash = hash(virtualNodeKey);
            hashRing.put(hash, node);
        }
        
        if (!physicalNodes.contains(node)) {
            physicalNodes.add(node);
        }
        
        log.info("添加物理节点 {} 到哈希环，创建 {} 个虚拟节点", 
            node.getIdentifier(), VIRTUAL_NODE_COUNT);
    }
    
    /**
     * 从哈希环移除物理节点
     */
    public void removePhysicalNode(PhysicalNode node) {
        for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
            String virtualNodeKey = node.getIdentifier() + "#VN" + i;
            long hash = hash(virtualNodeKey);
            hashRing.remove(hash);
        }
        
        physicalNodes.remove(node);
        
        log.info("从哈希环移除物理节点 {}", node.getIdentifier());
    }
    
    /**
     * 根据分片键查找目标节点
     * 
     * @param shardingKey 分片键
     * @return 物理节点
     */
    public PhysicalNode getNode(Long shardingKey) {
        if (hashRing.isEmpty()) {
            throw new IllegalStateException("哈希环为空，请先初始化");
        }
        
        long hash = hash(String.valueOf(shardingKey));
        
        // 顺时针查找第一个大于等于该hash的节点
        Map.Entry<Long, PhysicalNode> entry = hashRing.ceilingEntry(hash);
        
        // 如果没找到，说明hash值大于环上所有节点，返回第一个节点（环形）
        if (entry == null) {
            entry = hashRing.firstEntry();
        }
        
        return entry.getValue();
    }
    
    /**
     * 计算哪些数据需要从旧节点迁移到新节点
     * 
     * @param newNode 新添加的节点
     * @return 需要迁移的数据来源节点
     */
    public List<PhysicalNode> getDataMigrationSources(PhysicalNode newNode) {
        Set<PhysicalNode> sources = new HashSet<>();
        
        // 模拟新节点的虚拟节点，找到每个虚拟节点的前继节点
        for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
            String virtualNodeKey = newNode.getIdentifier() + "#VN" + i;
            long hash = hash(virtualNodeKey);
            
            // 找到该虚拟节点在环上的前继节点
            Map.Entry<Long, PhysicalNode> lowerEntry = hashRing.lowerEntry(hash);
            if (lowerEntry == null) {
                lowerEntry = hashRing.lastEntry();
            }
            
            if (lowerEntry != null && !lowerEntry.getValue().equals(newNode)) {
                sources.add(lowerEntry.getValue());
            }
        }
        
        return new ArrayList<>(sources);
    }
    
    /**
     * MD5哈希函数
     */
    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] digest = md.digest(key.getBytes(StandardCharsets.UTF_8));
            
            // 取前8个字节转换为long
            long hash = 0;
            for (int i = 0; i < 8; i++) {
                hash = (hash << 8) | (digest[i] & 0xFF);
            }
            return hash;
            
        } catch (NoSuchAlgorithmException e) {
            throw new RuntimeException("MD5算法不可用", e);
        }
    }
    
    /**
     * 物理节点
     */
    @Data
    public static class PhysicalNode {
        private final String database;
        private final String table;
        
        public PhysicalNode(String database, String table) {
            this.database = database;
            this.table = table;
        }
        
        /**
         * 节点唯一标识
         */
        public String getIdentifier() {
            return database + "." + table;
        }
        
        @Override
        public boolean equals(Object o) {
            if (this == o) return true;
            if (o == null || getClass() != o.getClass()) return false;
            PhysicalNode that = (PhysicalNode) o;
            return Objects.equals(database, that.database) && 
                   Objects.equals(table, that.table);
        }
        
        @Override
        public int hashCode() {
            return Objects.hash(database, table);
        }
    }
}
```

```java
            // 返回目标数据库

        }
        
        return allActualSplitDatabaseNames;
    }
```

### 2.3 扩容流程

```java
/**
 * 一致性哈希扩容服务
 */
@Service
@Slf4j
public class ConsistentHashScaleOutService {
    
    @Autowired
    private ConsistentHashingManager hashingManager;
    
    /**
     * 添加新节点并执行数据迁移
     */
    public void addNewNode(String database, String table) {
        PhysicalNode newNode = new PhysicalNode(database, table);
        
        // 1. 计算需要从哪些节点迁移数据
        List<PhysicalNode> sources = hashingManager.getDataMigrationSources(newNode);
        
        log.info("新节点 {} 需要从以下节点迁移数据: {}", 
            newNode.getIdentifier(), 
            sources.stream().map(PhysicalNode::getIdentifier).collect(Collectors.toList()));
        
        // 2. 执行数据迁移（异步）
        migrationExecutor.submit(() -> {
            migrateDataToNewNode(newNode, sources);
        });
        
        // 3. 添加新节点到哈希环（迁移完成后）
        // 注意：可以先添加到环中（新数据立即写入），旧数据慢慢迁移
        hashingManager.addPhysicalNode(newNode);
    }
    
    /**
     * 迁移数据到新节点
     */
    private void migrateDataToNewNode(PhysicalNode newNode, List<PhysicalNode> sources) {
        for (PhysicalNode source : sources) {
            // 遍历源节点的所有数据
            // 对每条数据重新计算hash，如果命中新节点，则迁移
            int pageSize = 1000;
            int offset = 0;
            
            while (true) {
                List<Order> orders = queryOrders(source, offset, pageSize);
                if (orders.isEmpty()) break;
                
                List<Order> toMigrate = orders.stream()
                    .filter(order -> {
                        PhysicalNode targetNode = hashingManager.getNode(order.getOrderNumber());
                        return targetNode.equals(newNode);
                    })
                    .collect(Collectors.toList());
                
                if (!toMigrate.isEmpty()) {
                    batchInsert(newNode, toMigrate);
                    log.info("从 {} 迁移 {} 条数据到 {}", 
                        source.getIdentifier(), toMigrate.size(), newNode.getIdentifier());
                }
                
                offset += pageSize;
            }
        }
    }
}
```

### 2.4 优缺点

#### ✅ 优点

1. **迁移量小**：只需迁移 1/N 的数据（N为节点总数）
2. **自动负载均衡**：虚拟节点确保数据均匀分布
3. **扩容平滑**：对现有节点影响最小

#### ❌ 缺点

1. **不支持范围查询**：哈希打散了数据的连续性
2. **实现复杂**：需要维护哈希环、虚拟节点
3. **迁移识别困难**：需要扫描所有数据判断是否需要迁移

## 方案3：幂次方扩容 + 按位分裂（⭐⭐⭐⭐⭐）

### 3.1 核心思想

**关键约束**：表数量必须是2的幂次方（2, 4, 8, 16, 32, ...）

```
原理：使用哈希值的低k位决定表号

初始：4表 (2^2)，使用低2位
  hash = ...1101 → 后2位 = 01 → 表1

扩容到8表 (2^3)，使用低3位
  hash = ...1101 → 后3位 = 101 → 表5
  
观察：
- 原表0 (00) 拆分为 新表0 (000) + 新表4 (100)
- 原表1 (01) 拆分为 新表1 (001) + 新表5 (101)
- 原表2 (10) 拆分为 新表2 (010) + 新表6 (110)
- 原表3 (11) 拆分为 新表3 (011) + 新表7 (111)

每个原表拆分为2个新表，只需迁移一半数据！
```

### 3.2 核心实现

#### 3.2.1 按位分裂算法

```java
package com.damai.shardingsphere;

import cn.hutool.core.collection.CollectionUtil;
import com.damai.enums.BaseCode;
import com.damai.exception.DaMaiFrameException;
import org.apache.shardingsphere.sharding.api.sharding.complex.ComplexKeysShardingAlgorithm;
import org.apache.shardingsphere.sharding.api.sharding.complex.ComplexKeysShardingValue;

import java.util.ArrayList;
import java.util.Collection;
import java.util.List;
import java.util.Map;
import java.util.Objects;
import java.util.Properties;

/**
 * 幂次方扩容的分表算法
 * 核心：表数量必须是2的幂次方，扩容时翻倍
 */
public class TableOrderPowerOf2Algorithm implements ComplexKeysShardingAlgorithm<Long> {
    
    /**
     * 当前表数量（必须是2的幂次方）
     */
    private int tableCount;
    
    /**
     * 用于计算表索引的位数
     */
    private int bitCount;
    
    @Override
    public void init(Properties props) {
        this.tableCount = Integer.parseInt(props.getProperty("table-count", "4"));
        
        // 验证是否为2的幂次方
        if (!isPowerOf2(tableCount)) {
            throw new IllegalArgumentException("表数量必须是2的幂次方，当前: " + tableCount);
        }
        
        // 计算位数：log2(tableCount)
        this.bitCount = Integer.numberOfTrailingZeros(tableCount);
    }
    
    @Override
    public Collection<String> doSharding(
        Collection<String> allActualSplitTableNames,
        ComplexKeysShardingValue<Long> complexKeysShardingValue) {
        
        List<String> actualTableNames = new ArrayList<>();
        String logicTableName = complexKeysShardingValue.getLogicTableName();
        Map<String, Collection<Long>> columnNameAndShardingValuesMap = 
            complexKeysShardingValue.getColumnNameAndShardingValuesMap();
        
        if (CollectionUtil.isEmpty(columnNameAndShardingValuesMap)) {
            return actualTableNames;
        }
        
        // 获取分片键
        Long shardingKey = getShardingKey(columnNameAndShardingValuesMap);
        
        if (Objects.nonNull(shardingKey)) {
            // 使用低bitCount位计算表索引
            int tableIndex = (int) (shardingKey & ((1L << bitCount) - 1));
            actualTableNames.add(logicTableName + "_" + tableIndex);
            return actualTableNames;
        }
        
        return allActualSplitTableNames;
    }
    
    /**
     * 判断是否为2的幂次方
     */
    private boolean isPowerOf2(int n) {
        return n > 0 && (n & (n - 1)) == 0;
    }
    
    private Long getShardingKey(Map<String, Collection<Long>> columnMap) {
        Collection<Long> orderNumberValues = columnMap.get("order_number");
        Collection<Long> userIdValues = columnMap.get("user_id");
        
        if (CollectionUtil.isNotEmpty(orderNumberValues)) {
            return orderNumberValues.stream().findFirst()
                .orElseThrow(() -> new DaMaiFrameException(BaseCode.ORDER_NUMBER_NOT_EXIST));
        } else if (CollectionUtil.isNotEmpty(userIdValues)) {
            return userIdValues.stream().findFirst()
                .orElseThrow(() -> new DaMaiFrameException(BaseCode.USER_ID_NOT_EXIST));
        }
        return null;
    }
}
```

#### 3.2.2 分库算法（类似）

```java
/**
 * 幂次方扩容的分库算法
 */
public class DatabaseOrderPowerOf2Algorithm implements ComplexKeysShardingAlgorithm<Long> {
    
    private int databaseCount;
    private int tableCount;
    private int tableBitCount;
    private int databaseBitCount;
    
    @Override
    public void init(Properties props) {
        this.databaseCount = Integer.parseInt(props.getProperty("database-count", "2"));
        this.tableCount = Integer.parseInt(props.getProperty("table-count", "4"));
        
        if (!isPowerOf2(databaseCount) || !isPowerOf2(tableCount)) {
            throw new IllegalArgumentException("库数和表数必须是2的幂次方");
        }
        
        this.tableBitCount = Integer.numberOfTrailingZeros(tableCount);
        this.databaseBitCount = Integer.numberOfTrailingZeros(databaseCount);
    }
    
    @Override
    public Collection<String> doSharding(
        Collection<String> allActualSplitDatabaseNames,
        ComplexKeysShardingValue<Long> complexKeysShardingValue) {
        
        Long shardingKey = getShardingKey(complexKeysShardingValue);
        
        if (shardingKey != null) {
            // 跳过表基因位，使用中间的databaseBitCount位
            int databaseIndex = (int) ((shardingKey >> tableBitCount) & 
                ((1L << databaseBitCount) - 1));
            
            String targetDatabase = "damai_order_" + databaseIndex;
            return Collections.singletonList(targetDatabase);
        }
        
        return allActualSplitDatabaseNames;
    }
    
    private boolean isPowerOf2(int n) {
        return n > 0 && (n & (n - 1)) == 0;
    }
}
```

### 3.3 扩容实施

#### 3.3.1 扩容迁移工具

```java
/**
 * 幂次方扩容迁移工具
 */
@Service
@Slf4j
public class PowerOf2ScaleOutService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    /**
     * 执行扩容（从oldCount扩容到newCount）
     * 
     * @param oldCount 旧表数量（必须是2的幂次方）
     * @param newCount 新表数量（必须是oldCount的2倍）
     */
    public void scaleOut(int oldCount, int newCount) {
        // 1. 验证参数
        if (newCount != oldCount * 2) {
            throw new IllegalArgumentException("新表数量必须是旧表数量的2倍");
        }
        
        if (!isPowerOf2(oldCount) || !isPowerOf2(newCount)) {
            throw new IllegalArgumentException("表数量必须是2的幂次方");
        }
        
        log.info("开始扩容：{} 表 → {} 表", oldCount, newCount);
        
        // 2. 创建新表
        for (int i = oldCount; i < newCount; i++) {
            createTable("damai_order_0", "d_order_" + i);
            createTable("damai_order_1", "d_order_" + i);
        }
        
        // 3. 按位分裂迁移数据
        for (int oldTableIndex = 0; oldTableIndex < oldCount; oldTableIndex++) {
            splitAndMigrate(oldTableIndex, oldCount, newCount);
        }
        
        log.info("扩容完成");
    }
    
    /**
     * 拆分单个旧表的数据
     * 
     * @param oldTableIndex 旧表索引
     * @param oldCount 旧表总数
     * @param newCount 新表总数
     */
    private void splitAndMigrate(int oldTableIndex, int oldCount, int newCount) {
        log.info("开始拆分表 {}", oldTableIndex);
        
        // 计算目标表索引
        // 原表 i 拆分为：新表 i 和 新表 (i + oldCount)
        int newTableIndex1 = oldTableIndex;           // 保持不变的部分
        int newTableIndex2 = oldTableIndex + oldCount; // 需要迁移的部分
        
        // 计算位数
        int oldBitCount = Integer.numberOfTrailingZeros(oldCount);
        int newBitCount = Integer.numberOfTrailingZeros(newCount);
        
        // 新增的那一位（第oldBitCount位）
        long newBitMask = 1L << oldBitCount;
        
        // 遍历旧表数据
        int pageSize = 1000;
        int offset = 0;
        long migratedCount = 0;
        
        while (true) {
            // 查询旧表数据（两个库都要查）
            List<Order> orders = queryOrdersFromTable(
                "damai_order_0", "d_order_" + oldTableIndex, offset, pageSize);
            orders.addAll(queryOrdersFromTable(
                "damai_order_1", "d_order_" + oldTableIndex, offset, pageSize));
            
            if (orders.isEmpty()) break;
            
            // 根据新位的值分组
            Map<Boolean, List<Order>> grouped = orders.stream()
                .collect(Collectors.partitioningBy(order -> {
                    // 检查orderNumber的第oldBitCount位是否为1
                    return (order.getOrderNumber() & newBitMask) != 0;
                }));
            
            // 新位为0的数据，保留在原表（不需要迁移）
            List<Order> stayInOldTable = grouped.get(false);
            
            // 新位为1的数据，迁移到新表
            List<Order> moveToNewTable = grouped.get(true);
            if (!moveToNewTable.isEmpty()) {
                // 插入到新表
                batchInsertToTable("damai_order_0", "d_order_" + newTableIndex2, moveToNewTable);
                
                // 从旧表删除
                batchDeleteFromTable("damai_order_0", "d_order_" + oldTableIndex, moveToNewTable);
                
                migratedCount += moveToNewTable.size();
            }
            
            offset += pageSize;
            log.info("表 {} 已迁移 {} 条数据", oldTableIndex, migratedCount);
        }
        
        log.info("表 {} 拆分完成，共迁移 {} 条数据到表 {}", 
            oldTableIndex, migratedCount, newTableIndex2);
    }
    
    /**
     * 创建表
     */
    private void createTable(String database, String table) {
        String sql = String.format(
            "CREATE TABLE %s.%s LIKE %s.d_order_0",
            database, table, database
        );
        jdbcTemplate.execute(sql);
        log.info("创建表：{}.{}", database, table);
    }
    
    /**
     * 批量插入
     */
    private void batchInsertToTable(String database, String table, List<Order> orders) {
        String sql = String.format(
            "INSERT INTO %s.%s (order_number, user_id, ...) VALUES (?, ?, ...)",
            database, table
        );
        
        jdbcTemplate.batchUpdate(sql, new BatchPreparedStatementSetter() {
            @Override
            public void setValues(PreparedStatement ps, int i) throws SQLException {
                Order order = orders.get(i);
                ps.setLong(1, order.getOrderNumber());
                ps.setLong(2, order.getUserId());
                // ... 其他字段
            }
            
            @Override
            public int getBatchSize() {
                return orders.size();
            }
        });
    }
    
    /**
     * 批量删除
     */
    private void batchDeleteFromTable(String database, String table, List<Order> orders) {
        String sql = String.format(
            "DELETE FROM %s.%s WHERE order_number = ?",
            database, table
        );
        
        jdbcTemplate.batchUpdate(sql, new BatchPreparedStatementSetter() {
            @Override
            public void setValues(PreparedStatement ps, int i) throws SQLException {
                ps.setLong(1, orders.get(i).getOrderNumber());
            }
            
            @Override
            public int getBatchSize() {
                return orders.size();
            }
        });
    }
    
    private boolean isPowerOf2(int n) {
        return n > 0 && (n & (n - 1)) == 0;
    }
    
    private List<Order> queryOrdersFromTable(
        String database, String table, int offset, int limit) {
        String sql = String.format(
            "SELECT * FROM %s.%s LIMIT %d, %d",
            database, table, offset, limit
        );
        return jdbcTemplate.query(sql, new BeanPropertyRowMapper<>(Order.class));
    }
}
```

### 3.4 优缺点

#### ✅ 优点

1. **迁移量固定**：每次扩容只需迁移50%数据
2. **实现简单**：纯位运算，逻辑清晰
3. **性能高**：无需额外查表或计算复杂哈希
4. **可预测**：扩容规模和迁移量都可提前计算

#### ❌ 缺点

1. **扩容限制**：只能按2的幂次方扩容（2→4→8→16）
2. **灵活性差**：不能自由增减节点
3. **仍需迁移**：虽然只迁移50%，但数据量大时仍然耗时

## 总结

### 三种方案的本质区别

1. **方案1（虚拟分片）**：增加一层间接层，解耦逻辑分片和物理分片

3. **方案3（幂次方扩容）**：利用二进制位的天然分裂特性，精确控制迁移范围
