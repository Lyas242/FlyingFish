## 🎯 核心问题

**问题**：订单号是用2库×4表的规则生成的，扩容到2库×8表后，旧订单号还能正确查询吗？

**答案**：✅ **能查到！** 

# 一、📖 订单号基因的秘密

## 1.1 订单号的结构

```
完整订单号（64位）：
[时间戳41位][数据中心5位][机器5位][序列12位][基因N位]
                                              ↑
                                    这部分是关键！
```

**基因位**：订单号的**最后N位**，决定了订单存在哪个库、哪个表。

## 1.2 生成基因的代码（逐行讲解）

### 1.2.1 扩容前：2库×4表

```java
// ========== 配置 ==========
databaseCount = 2;  // 2个库
tableCount = 4;     // 4个表

// ========== 步骤1：计算需要几位基因 ==========
long tableGeneLength = log2N(4);      // log2(4) = 2位
long databaseGeneLength = log2N(2);   // log2(2) = 1位
long totalGeneLength = 3;             // 2 + 1 = 3位

// 为什么？
// 4个表需要2位才能表示：00, 01, 10, 11（0-3）
// 2个库需要1位才能表示：0, 1（0-1）
// 所以总共需要 3 位基因

// ========== 步骤2：创建掩码 ==========
long geneMask = (1L << 3) - 1;

// 详细计算过程：
// (1L << 3) = 1000（二进制）= 8（十进制）
// 8 - 1 = 7（十进制）= 0111（二进制）

// geneMask = 0111（二进制）= 7（十进制）
// 
// 这个掩码的作用：提取后3位
// 为什么？因为掩码的后3位都是1，前面都是0

// ========== 步骤3：从userId提取后3位 ==========
userId = 13;  // 假设用户ID是13

// 13 的二进制是多少？
// 13 = 8 + 4 + 1 = 1101（二进制）

long userGene = userId & geneMask;
// userGene = 1101 & 0111
//          = 0101（只保留了后3位）
//          = 5（十进制）

// ========== 结果 ==========
// 订单号的最后3位 = 101
```

### 1.2.2 图解：掩码的作用

```
userId = 13 = 1101（假设是4位）

掩码 geneMask = 0111

按位与：
  1101  (userId = 13)
& 0111  (geneMask，只保留后3位)
-------
  0101  (结果 = 5，就是后3位)
   ↑↑↑
   这就是基因！
```

# 二、扩容前的路由计算

## 2.1 扩容前的配置

```
2库×4表：
- 库：damai_order_0, damai_order_1
- 表：d_order_0, d_order_1, d_order_2, d_order_3

需要：
- 2位表示表（00, 01, 10, 11 → 0,1,2,3）
- 1位表示库（0, 1）
- 总共3位基因
```

## 2.2 订单号：末尾 101

```
订单号：...末尾101
（为了简化，我们只关注最后3位基因）
```

## 2.3 如何计算表索引？

### 2.3.1 代码

```java
// 旧算法（4表）
int tableCount = 4;
long tableIndex = orderNumber & (tableCount - 1);
```

### 2.3.2 详细计算

```
步骤1：计算掩码
tableCount - 1 = 4 - 1 = 3 = 011（二进制）

为什么是 011？
因为 4 = 100（二进制）
100 - 1 = 011

步骤2：提取后2位
orderNumber = ...101
掩码         = ...011

按位与：
  101
& 011
-----
  001  = 1（十进制）

结果：tableIndex = 1
```

### 2.3.3 图解

```
订单号末尾：1 0 1
          ↑ ↑ ↑
     第3位 第2位 第1位

提取后2位（表索引）：
掩码 011 只保留后2位

101 & 011 = 01 = 1

表索引 = 1 → d_order_1
```

## 2.4 如何计算库索引？

### 2.4.1 代码

```java
// 旧算法（2库×4表）
long tableGeneLength = 2;  // 表用了2位
int databaseCount = 2;

long dbIndex = (orderNumber >> tableGeneLength) & (databaseCount - 1);
```

### 2.4.2 详细计算

```
步骤1：右移2位，跳过表基因
orderNumber = 101
右移2位：101 >> 2 = 1

为什么右移2位？
因为后2位是表基因，我们要跳过它，获取第3位（库基因）

101（3位）
→ 10（右移1位，去掉第1位）
→ 1（右移2位，去掉前2位）

步骤2：提取库基因
databaseCount - 1 = 2 - 1 = 1 = 01（二进制）

1 & 01 = 1

结果：dbIndex = 1
```

### 2.4.3 图解

```
订单号末尾：1 0 1
          ↑ ↑ ↑
         库 表 表
       基因 基因 基因
   (第3位) (第2位) (第1位)

计算库索引：
1. 右移2位，跳过表基因：101 >> 2 = 1
2. 提取第1位（库基因）：1 & 1 = 1

库索引 = 1 → damai_order_1
```

## 2.5 最终位置

```
订单号末尾：101

表索引 = 01（后2位）= 1
库索引 = 1（第3位）= 1

存储位置：damai_order_1.d_order_1 ✅
```

# 三、🚀 扩容后会怎样？

## 3.1 扩容后的配置

```
2库×8表：
- 库：damai_order_0, damai_order_1（不变）
- 表：d_order_0 ~ d_order_7（翻倍）

需要：
- 3位表示表（000~111 → 0~7）
- 1位表示库（0, 1）
- 总共4位基因
```

## 3.2 新算法计算（同样的订单号 101）

### 3.2.1 计算表索引

```java
// 新算法（8表）
int tableCount = 8;
long tableIndex = orderNumber & (tableCount - 1);
```

```
步骤1：计算掩码
tableCount - 1 = 8 - 1 = 7 = 0111（二进制）

为什么是 0111？
因为 8 = 1000
1000 - 1 = 0111

步骤2：提取后3位
orderNumber = ...101
掩码        = ...0111

按位与：
  0101
& 0111
------
  0101 = 5（十进制）

结果：tableIndex = 5
```

#### 3.2.2 计算库索引

```java
// 新算法（2库×8表）
long tableGeneLength = 3;  // 表现在用3位
int databaseCount = 2;

long dbIndex = (orderNumber >> tableGeneLength) & (databaseCount - 1);
```

```
步骤1：右移3位
orderNumber = 101（只有3位）
右移3位：101 >> 3 = 0

解释：
101 只有3位，右移3位后所有位都没了，变成0

步骤2：提取库基因
0 & 1 = 0

结果：dbIndex = 0
```

## 3.3 新算法的期望位置

```
订单号末尾：101（旧订单号）

新算法计算：
- 表索引 = 101 & 111 = 101 = 5
- 库索引 = (101 >> 3) & 1 = 0

新算法期望位置：damai_order_0.d_order_5
```

## 3.4 问题出现了！

```
旧位置：damai_order_1.d_order_1（数据还在这里）
新位置：damai_order_0.d_order_5（新算法期望这里）

数据在旧位置，新算法查新位置 → 找不到！❌
```

# 四、✅ 解决方案 - 按位分裂迁移

## 4.1 核心思想

**迁移数据，让数据的实际位置 = 新算法的期望位置**

```
订单号末尾 101 的数据：
从 damai_order_1.d_order_1
迁移到 damai_order_0.d_order_5

迁移后：
- 数据位置：damai_order_0.d_order_5
- 新算法查：damai_order_0.d_order_5
- 结果：✅ 能查到！
```

## 4.2 迁移判断：检查"新增位"

### 什么是新增位？

```
旧算法（4表）：用后2位
新算法（8表）：用后3位

对比：
旧：× × 0 1  （只用后2位）
新：× 1 0 1  （用后3位）
     ↑
这是新增的第3位！
```

#### 新增位的作用

```
原表1（后2位 = 01）中的数据：

数据A：末尾 001
       后2位 = 01 ✓（在表1）
       第3位 = 0

数据B：末尾 101
       后2位 = 01 ✓（在表1）
       第3位 = 1

扩容后：
- 数据A（第3位=0）：新算法 001 & 111 = 001 = 1 → 留在表1
- 数据B（第3位=1）：新算法 101 & 111 = 101 = 5 → 迁移到表5
```

## 4.3 如何检查第3位？

### 方法：创建第3位的掩码

```java
long newBitMask = 1L << 2;  // 创建第3位的掩码
```

### 详细计算

```
步骤1：1 << 2
1 的二进制：  0001
左移2位：     0100

结果：0100 = 4（十进制）

步骤2：理解掩码 0100
0100 的含义：
  位置：  第4位 第3位 第2位 第1位
  数值：   0    1    0    0
               ↑
          只有第3位是1

这个掩码用来检查第3位！
```

### 使用掩码检查

```java
if ((orderNumber & newBitMask) != 0) {
    // 第3位是1，需要迁移
}
```

```
示例1：订单号末尾 001

  001
& 100  (newBitMask)
-----
  000 = 0

结果：第3位=0，不迁移
```

```
示例2：订单号末尾 101

  101
& 100  (newBitMask)
-----
  100 ≠ 0

结果：第3位=1，需要迁移！
```

## 4.4 完整的迁移逻辑

```java
// 遍历原表1的所有数据
for (Order order : d_order_1的所有订单) {
    // 创建第3位的掩码
    long newBitMask = 1L << 2;  // 0100
    
    // 检查订单号的第3位
    if ((order.getOrderNumber() & newBitMask) != 0) {
        // 第3位=1，需要迁移
        // 用新算法计算目标位置
        long newTableIndex = order.getOrderNumber() & 0b111;  // 提取后3位
        long newDbIndex = (order.getOrderNumber() >> 3) & 0b1;
        
        // 迁移到新位置
        迁移(order, "damai_order_" + newDbIndex, "d_order_" + newTableIndex);
    } else {
        // 第3位=0，留在原位置
        保持不动;
    }
}
```

# 五、🎯 为什么旧订单号能查到？

## 5.1 完整流程

```
【扩容前】
订单号：...101
算法：4表（后2位）
位置：damai_order_1.d_order_1 ✅

【扩容迁移】
检查第3位：101 & 100 = 100 ≠ 0
判断：第3位=1，需要迁移
新算法计算：101 & 111 = 5，库=(101>>3)&1=0
迁移到：damai_order_0.d_order_5 ✅

【扩容后查询】
订单号：...101（同一个订单号）
新算法：8表（后3位）
计算：101 & 111 = 5，库=(101>>3)&1=0
期望位置：damai_order_0.d_order_5
实际位置：damai_order_0.d_order_5（已迁移）
结果：✅ 能查到！
```

## 5.2 关键原理

```
秘诀：按位分裂

4表→8表，用的位数从2位→3位

原表拆分规律：
原表0 (00) → 新表0 (000) + 新表4 (100)
原表1 (01) → 新表1 (001) + 新表5 (101)
原表2 (10) → 新表2 (010) + 新表6 (110)
原表3 (11) → 新表3 (011) + 新表7 (111)

观察：
- 后2位相同 → 原来在同一个表
- 第3位决定去向：
  * 第3位=0 → 留在原表（表号不变）
  * 第3位=1 → 迁移到新表（表号+4）
```

## 5.3 图解：原表1的拆分

```
原表1（后2位=01）的数据：

├─ 末尾 001（第3位=0）
│   新算法：001 & 111 = 001 = 1
│   去向：表1（不变）
│
├─ 末尾 101（第3位=1）
│   新算法：101 & 111 = 101 = 5
│   去向：表5（1+4=5）
│
└─ 末尾 201？不存在！
    因为后2位必须是01，第3位只能是0或1
    所以只有 001 和 101 两种可能

结论：原表1拆分为新表1和新表5
```

# 六、💡 核心总结

## 6.1 三个关键点

### 1. 订单号的低位是基因

```
订单号末尾的N位决定了数据存在哪里
扩容前后，这N位的值不变（都是101）
```

### 2. 按位分裂迁移

```
新算法用更多位（2位→3位）
新增的位（第3位）决定数据去向
通过检查新增位，决定是否迁移
```

### 3. 迁移后数据在新算法期望的位置

```
旧订单号 101：
- 旧算法期望：表1
- 迁移后位置：表5
- 新算法期望：表5 ✅

完美匹配！所以能查到！
```

## 6.2 为什么必须是2的幂次方？

### 原因：二进制位的增长

```
4表（2²）：用2位 → 00, 01, 10, 11
8表（2³）：用3位 → 000, 001, 010, 011, 100, 101, 110, 111

关系：
新3位的前2位 = 旧2位 ✓
这种"位对齐"只有在幂次方时才成立！
```

### 反例：4表→5表（不是幂次方）

```
5表不是2的幂次方，无法用简单的位运算

4 = 100（2²）
5 = 101（不是2的幂次方）

5-1 = 4 = 100

提取：orderNumber & 100 只能得到第3位
无法表示0-4的5个值

必须用取模：orderNumber % 5
但取模没有"按位分裂"的规律！
```

## 6.3 一句话总结

**扩容时通过检查订单号的新增位，将数据迁移到新算法期望的位置，所以旧订单号用新算法查询能找到数据。**

# 八、🎓 常见问题

## Q1：新订单号（扩容后生成）会怎样？

**A**：新订单号用新算法（4位基因），直接路由到正确位置。

```
扩容后，userId=13 生成新订单号：
- userGene = 13 & 0b1111 = 0b1101 = 13
- 订单号末尾：1101（4位）

新算法计算：
- 表索引：1101 & 0111 = 0101 = 5
- 库索引：(1101 >> 3) & 1 = 1

直接写入：damai_order_1.d_order_5 ✅
```

## Q2：迁移过程中如果有新订单怎么办？

**A**：有两种方案：

### 方案A：停写扩容（推荐）

```
1. 停止订单服务
2. 执行迁移
3. 更新配置，重启
4. 开放服务
```

### 方案B：双写扩容（复杂但零停机）

```
1. 新订单同时写旧表和新表
2. 后台迁移旧数据
3. 验证完成后切换
```

## Q3：如何验证迁移正确性？

**A**：遍历所有表，检查每条订单是否在新算法期望的位置。

```java
for (每个表的每条订单) {
    long expectedTableIndex = orderNumber & 0b111;
    long expectedDbIndex = (orderNumber >> 3) & 0b1;
    
    if (实际位置 != 期望位置) {
        报错！
    }
}
```
