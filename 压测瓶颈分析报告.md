## 🚨 **核心问题确认**

### ✅ **确认是单机部署资源竞争问题**

通过深入分析，**100%确认**这是单机部署导致的资源竞争问题：

#### 1. **线程数过载**
- 系统总线程数: **1,140+个**
- 单个服务最高线程数: **1,090个**（Program Service）
- 线程上下文切换开销巨大

#### 2. **CPU资源竞争激烈**
- Program Service CPU使用率达到**122%**（超过单核100%）
- 多个Java进程同时争夺CPU资源
- 系统负载持续在**5.0+**（理想值应<2.0）

#### 3. **内存压力临界**
- 总内存使用: **63GB**（接近系统极限）
- 多个JVM堆内存同时分配
- GC压力导致系统卡顿

#### 4. **网络连接数过多**
- ESTABLISHED连接: **1,500+个**
- 单机承载所有服务的网络连接
- 网络IO竞争

## 📊 **瓶颈根因分析**

### 主要瓶颈：Program Service (PID 15132)
```
服务名称: program-service
CPU使用: 75-122%（严重过载）
线程数: 1,090个（远超正常值）
内存: 2.1GB
状态: 严重瓶颈，是响应缓慢的直接原因
```

**为什么Program Service成为瓶颈？**
1. **业务复杂度高**: 节目订单创建涉及复杂业务逻辑
2. **数据库操作密集**: 分库分表查询、座位锁定、库存扣减
3. **缓存操作频繁**: Redis读写、分布式锁竞争
4. **线程池配置过高**: 创建了过多线程争夺CPU
5. **分布式锁串行化**: 同一节目的请求必须串行处理

### 次要瓶颈：监控和压测工具
```
JMeter: 554线程，波动性CPU使用
SkyWalking: 321线程，持续5%CPU + 3.2GB内存
其他IDEA编译进程: 额外的资源消耗
```

## 💡 **解决方案优先级**

### 🔥 **立即执行（高优先级）**

#### 1. 关闭非必要服务
```bash
# 临时关闭SkyWalking（节省5%CPU + 3.2GB内存）
# 关闭IDEA编译进程
# 只保留核心业务服务
```

#### 2. 降低JMeter并发数
```
当前: 500并发用户
建议: 降至50-100并发用户
逐步增加: 50 → 100 → 200 → 300
```

#### 3. 优化Program Service配置
```yaml
# 大幅降低线程池配置
server:
  tomcat:
    threads:
      max: 50              # 从当前800降至50
      min-spare: 10        # 从100降至10
    max-connections: 500   # 从12000降至500

# 降低数据库连接池
hikari:
  maximum-pool-size: 10  # 从100降至10
  minimum-idle: 2        # 从30降至2
```

### ⚡ **中期优化（中优先级）**

#### 1. 业务逻辑优化
- 移除或简化分布式锁
- 优化数据库查询逻辑
- 减少Redis操作次数

#### 2. JVM参数调优
```bash
# Program Service JVM优化
-Xms2g -Xmx2g          # 降低堆内存
-XX:+UseG1GC           # 使用G1GC
-XX:MaxGCPauseMillis=50 # 降低GC暂停时间
```

### 🏗️ **长期解决（低优先级）**

#### 1. 分离部署架构
```
机器A: Program Service + Order Service
机器B: MySQL + Redis
机器C: Kafka + Zookeeper + Nacos
机器D: JMeter压测工具
```
