```

→ 虚拟分片266 → 库0.表2
```

**现象**：虚拟分片ID不同（266 vs 258），但最终路由到相同位置（库0.表2）！

## 核心机制解析

### 为什么 physicalShardIndex 相同？

**关键：订单号生成时嵌入了用户ID的基因位！**

回顾 `SnowflakeIdGenerator.getOrderNumber()` 方法：

```java
public synchronized long getOrderNumber(long userId, 
                                        long tableCount, 
                                        long databaseCount) {
    // 步骤1：计算基因位数
    long tableGeneLength = log2N(tableCount);        // 2位
    long databaseGeneLength = log2N(databaseCount);  // 1位
    long totalGeneLength = tableGeneLength + databaseGeneLength; // 3位

    // 步骤2：从userId中提取基因位（后3位）
    long geneInfo = userId & ((1L << totalGeneLength) - 1);

    // 步骤3：生成订单编号（将userId的基因位嵌入到订单号的后3位）
    return ((timestamp - BASIS_TIME) << timestampLeftShift)
            | (datacenterId << datacenterIdShift)
            | (workerId << workerIdShift)
            | (sequence << totalGeneLength)
            | geneInfo;  // ← 关键：嵌入userId的基因位
}
```

**结果**：
```
userId的后3位 = 订单号的后3位

验证：
  userId:      1136344580112367618 & 0b111 = 2
  orderNumber: 1978005150261182474 & 0b111 = 2
  ✅ 相同！
```

## 🔑 核心问题：为什么 266 和 258 都在 [256, 383] 范围内？

### 💡 最核心的本质（一句话说清楚）

**关键就是这一行代码**：

```java
int virtualOffset = (int)(Math.abs(shardingKey) % virtualShardsPerPhysical);
//                                               ↑
//                                          对128取模
```

**核心逻辑**：

1. **shardingKey** 可以是订单号（1978005150261182474）或用户ID（1136344580112367618）
2. **对128取模**：`shardingKey % 128`，结果必然是 **0-127** 之间
3. **计算虚拟分片ID**：
   ```
   logicalShardId = physicalShardIndex × 128 + virtualOffset
                  = physicalShardIndex × 128 + (0~127之间的数)
   ```
4. **因为 physicalShardIndex 相同**（都是2，通过基因法保证）：
   ```
   logicalShardId 必然在 [2×128, 2×128+127] = [256, 383] 范围内
   ```
5. **最终分库分表的位置也就是相同的**（都路由到 库0.表2）

**就是这么个意思！✅**

## 数学保证

### 核心公式

```
logicalShardId = physicalShardIndex × 128 + virtualOffset
```

其中：
- `virtualOffset` 的范围：**0-127**（因为 `shardingKey % 128`）
- 只要 `physicalShardIndex` 相同，`logicalShardId` 就一定在同一个128范围内！

### 应用到实际数据

```
订单号：logicalShardId = 2 × 128 + 10 = 256 + 10 = 266
用户ID：logicalShardId = 2 × 128 + 2  = 256 + 2  = 258
```

**分析**：
- 订单号的虚拟偏移是 10（在0-127范围内）
- 用户ID的虚拟偏移是 2（在0-127范围内）
- 两者的起始位置都是 2×128 = 256
- 所以最终结果都在 [256, 256+127] = [256, 383] 范围内

### 反向验证

```
266 ÷ 128 = 2 余 10
  → 物理分片索引 = 2
  → 虚拟偏移 = 10
  → 范围 = [2×128, 2×128+127] = [256, 383]
  → 266 在范围内 ✓

258 ÷ 128 = 2 余 2
  → 物理分片索引 = 2
  → 虚拟偏移 = 2
  → 范围 = [2×128, 2×128+127] = [256, 383]
  → 258 在范围内 ✓
```

### 数学铁律

**只要 `physicalShardIndex` 相同，虚拟分片ID就一定在同一个128范围内：**

```
[physicalShardIndex × 128, physicalShardIndex × 128 + 127]
```

**原因**：
- `virtualOffset = shardingKey % 128`，范围固定是 **0-127**
- `logicalShardId = physicalShardIndex × 128 + virtualOffset`
- 最小值：`physicalShardIndex × 128 + 0`
- 最大值：`physicalShardIndex × 128 + 127`
- 所以范围必然是 **[起始, 起始+127]**

## 🤔 常见疑问解答（FAQ）

### Q1: 为什么虚拟分片ID不同，但能路由到同一个表？

**A**: 因为系统设计是 **128个虚拟分片 → 1个物理表**！

```
就像一栋楼有128个房间（虚拟分片），但所有房间都在同一栋楼（物理表）里！

虚拟分片258 → 在第2栋楼的第2个房间
虚拟分片266 → 在第2栋楼的第10个房间

两个房间号不同，但都在第2栋楼里！
```

### Q2: 为什么订单号和用户ID的虚拟分片ID不同？

**A**: 因为它们的 **虚拟偏移（virtualOffset）** 不同！

```
virtualOffset = shardingKey % 128

订单号：1978005150261182474 % 128 = 10
用户ID：1136344580112367618 % 128 = 2

虚拟偏移不同 → 虚拟分片ID不同
```

但是！它们的 **物理分片索引（physicalShardIndex）** 相同（都是2），所以都在同一个128范围内（256-383）！

### Q3: 怎么保证订单号和用户ID的物理分片索引相同？

**A**: 订单号生成时 **嵌入了用户ID的基因位**！

```java
// 在生成订单号时
geneInfo = userId & 0b111;  // 提取userId的后3位
orderNumber = ... | geneInfo;  // 嵌入到订单号的后3位

// 所以
userId的后3位 = orderNumber的后3位 = 基因位 = 2
```

基因位相同 → 物理分片索引相同 → 虚拟分片在同一个128范围内！

### Q4: 为什么一定是128个虚拟分片一组？

**A**: 这是由系统参数决定的：

```
虚拟分片总数 = 1024（固定）
物理分片数 = 2库 × 4表 = 8
每组虚拟分片数 = 1024 ÷ 8 = 128

如果是2库×8表（16个物理分片）：
  每组虚拟分片数 = 1024 ÷ 16 = 64

如果是4库×4表（16个物理分片）：
  每组虚拟分片数 = 1024 ÷ 16 = 64
```

**公式**：`每组大小 = 1024 ÷ (库数 × 表数)`

### Q5: 如果只迁移部分虚拟分片会怎样？

**A**: 会导致查询不一致！

```
错误示例：
  虚拟分片258 → 迁移到新表 table_6
  虚拟分片266 → 留在旧表 table_2
```
